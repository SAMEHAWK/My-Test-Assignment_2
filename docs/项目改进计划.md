## 一、文档目的与范围

### 1.1 目的

基于当前仓库实现与外部评审反馈，整理**以代码架构与机制可靠性为主**的改进项，形成可分批执行、可验收的改进路线图。

本文档**不是**新需求规格；改进方向与 `docs/架构设计.md` 一致，重点补齐：

- 机制边界与数据一致性
- 架构可验证性（测试、清单、Git）
- 交付专业度（命名、Prefab 体系、文档体量）
- 长期可维护性（资源生命周期、依赖方式）

### 1.2 不在范围内

按题面要求，以下** deliberately 不做**：

- 美术升级（材质、模型、特效、UI 美化）
- 新增玩法系统（战斗、任务、存档等）
- 升级 Unity 版本或新增 Package（除非某项改进明确依赖且经确认）
- 大规模重写（如全面 DI 框架、ECS 迁移）

## 四、分项改进说明

### IMP-01 — 生产在途锁与产出失败回滚 【P0】

**问题**

`BuildingProduction.TryProduce()` 在通过 `CanFitOutputRecipe()` 后立即扣除原料并启动产出动画；下一轮生产仍只检查 `outputWarehouse.GetTotalStored()`，**不统计正在飞行、尚未入库的产出**。

产出回调片段（当前逻辑）：

```csharp
if (capturedOutput != null && capturedOutput.CanAddResource(capturedType, 1))
    capturedOutput.AddResource(capturedType, 1);
// else: 静默失败，原料已扣
```

**影响**

- Output 接近满库时，可能出现「双开工 → 一个产出落地失败 → 资源丢失」
- 属于机制/架构严谨性问题，与画面无关

**建议方案**

1. 在 `BuildingProduction` 增加 `_inFlightOutputCount`（或按 `ResourceType` 计数）。
2. `CanFitOutputRecipe()` 改为：`GetTotalStored() + _inFlightOutputCount + 本轮产出 ≤ capacity`。
3. 每发起一笔产出动画：` _inFlightOutputCount++`；回调成功：`--` 并 `AddResource`；回调失败：`--` 并 **回滚对应原料** 或至少 `Debug.LogError` + 状态切 `OutputFull`。
4. 可选：生产进行中设 `_isProductionCycleActive`，同一 Tick 内禁止重入 `TryProduce()`。

**涉及文件**

- `Assets/Scripts/Building/BuildingProduction.cs`
- `docs/架构设计.md`（库存时序 / 验证清单 — 实施后更新）

**验收标准**

- 建筑 1 Output 容量 10、间隔 3s：故意不取货直到将满，连续观察 30s，库存 + 在途总数不超过 capacity，无 Console Error。
- 手动将 Output capacity 改为 1，连续两轮生产，第二次不应丢 N1（建筑 1 无 input）或不应 silently fail（建筑 2/3 有 input 时应回滚或停产）。

**风险**

- 回滚原料时需与已播放的消耗动画语义一致（消耗动画已是纯视觉，回滚数据即可）。

### IMP-11 — Singleton 依赖可选注入（非全面 DI） 【P2】

**问题**

`WarehouseView`、`BackpackView`、`TransferManager` 等直接访问 `ResourceObjectPool.Instance` / `TransferManager.Instance`。

**建议方案（最小改动）**

1. 保持 Instance 作为 Play 模式默认入口。
2. View 与 Production 已有 `[SerializeField]` 可选引用模式（如 `transferManager`），扩展到 `ResourceObjectPool`：
   - `[SerializeField] ResourceObjectPool objectPool;`
   - 私有属性 `Pool => objectPool != null ? objectPool : ResourceObjectPool.Instance;`
3. 在 README 或架构文档「后续扩展」中说明：团队扩大后可引入 Scene 级 `GameServices` 挂载点，而非立刻上 Zenject 等框架。

**涉及文件**

- `Assets/Scripts/Transfer/ResourceObjectPool.cs`（无需改 singleton）
- `Assets/Scripts/Building/WarehouseView.cs`
- `Assets/Scripts/Player/BackpackView.cs`
- `Assets/Scripts/Transfer/TransferManager.cs`

**验收标准**

- 场景中显式拖引用时不再依赖 Instance 查找；留空时行为与现版一致。

Calls to singletons are scattered directly throughout the code, instead of being cached in fields during initialization. The reason for processing interruptions is not reflected in gameplay.*

以下逐条对照**当前代码**（含 IMP-01～11 之后的状态），说明评审为何成立、与第一轮改进的关系、以及第二轮应做什么。

---

### 反馈 1：库存逻辑重复（Warehouse ↔ PlayerBackpack）

**评审在说什么**

「Recycler's warehouse」对应本项目的 **`Warehouse`**（建筑 Input/Output 仓库）；「Player's backpack」对应 **`PlayerBackpack`**。二者各自实现了一套几乎相同的「按类型计数 + 容量上限 + 增删查」逻辑。

**代码证据 — 重复的核心部分**

| 能力 | `Warehouse.cs` | `PlayerBackpack.cs` | 重复度 |
|------|----------------|---------------------|--------|
| 存储结构 | `Dictionary<ResourceType, int> storedResources` | 同左 | 完全相同 |
| 添加 | `CanAddResource` → `AddResource` | 同左（默认 amount=1） | ~95% |
| 移除 | `RemoveResource` + 归零删 key | 同左（实现细节略不同） | ~90% |
| 总量/满库 | `GetTotalStored` / `IsFull` | 同左 | 完全相同 |
| 调试串 | `RefreshDebugDisplay` + StringBuilder | 同左 | 完全相同 |

`Warehouse` 独有能力：`AcceptsType` / `acceptedTypes`、`WarehouseType`、锚点、`OnResourceAdded(type, amount)`。

`PlayerBackpack` 独有能力：`backpackAnchor`、粗粒度 `OnBackpackChanged`（无类型参数）。

**架构上的问题**

- 文档虽写「Model-View 分离」，但**缺少共享的 Domain 层**；两个 MonoBehaviour 各写一遍库存，属于典型的 **Copy-Paste 架构**。
- 后续若改规则（例如按类型分格容量、堆叠上限），必须改两处且易漏测。
- EditMode 测试（IMP-04）也分别测两个类，进一步证明逻辑未统一。

**合理改法（→ IMP-12）**

```
Assets/Scripts/Core/ResourceInventory.cs   ← 纯 C# 类，不含 MonoBehaviour
    ├── Warehouse.cs          ← 组合 ResourceInventory + 仓库特有字段/事件
    └── PlayerBackpack.cs     ← 组合 ResourceInventory + 背包锚点/事件
```

**与第一轮改进的关系**：IMP-04 增加了测试，但**加深了重复**（两套测试测同一套逻辑）；未解决根因。

---

### 反馈 2：Singleton 调用分散，未在初始化时缓存

**评审在说什么**

不是单纯「用了 Singleton」，而是：

1. 多处直接（或间接）访问 `TransferManager.Instance` / `ResourceObjectPool.Instance`；
2. 引用应在 **`Awake` / `Start` 解析一次并写入字段**，而不是在 Update、协程、每次 `Get/Return` 时通过属性再查 Instance。

**代码证据 — 当前模式（IMP-11 之后仍如此）**

以下类均使用 **lazy 属性**，每次访问都可能触发 `Instance` 查找：

| 文件 | 模式 |
|------|------|
| `BuildingProduction.cs` | `Transfer => transferManager ?? TransferManager.Instance` |
| `PlayerWarehouseInteraction.cs` | 同上 |
| `TransferManager.cs` | `Pool => objectPool ?? ResourceObjectPool.Instance` |
| `WarehouseView.cs` / `BackpackView.cs` | 同上 |

示例（`BuildingProduction` 协程每段动画间隔都会读 `Transfer`）：

```csharp
// 每次 PlayAnimationsStaggered 迭代都经属性访问 Instance
private TransferManager Transfer => transferManager != null ? transferManager : TransferManager.Instance;
```

**与 IMP-11 的差距**

IMP-11 增加了 `[SerializeField] optional` 引用，但：

- 未在 `Awake()` 中 `_transfer = transferManager != null ? transferManager : TransferManager.Instance;`
- View 在 `RefreshView()` 循环里多次 `Pool?.Get()` / `Pool?.Return()`，每次经过属性解析
- 评审期望的是 **依赖在生命周期早期固定**，便于测试注入、空引用排查和性能可预期

**合理改法（→ IMP-13）**

```csharp
private TransferManager _transfer;

private void Awake()
{
    _transfer = transferManager != null ? transferManager : TransferManager.Instance;
    if (_transfer == null)
        Debug.LogError("[BuildingProduction] TransferManager not found.", this);
}
```

全项目统一：`BuildingProduction`、`PlayerWarehouseInteraction`、`TransferManager`、`WarehouseView`、`BackpackView` 在初始化阶段缓存，后续只读字段。

**可选进阶**（本轮不必须）：Scene 级 `GameServices` 空物体统一挂载 Manager，Awake 注入，彻底去掉 `Instance` 静态访问。

### 反馈 3：加工中断原因未反映到玩法（Gameplay）

**评审在说什么**

「Processing interruptions」= 建筑停产状态：`InputMissing`（缺料）、`OutputFull`（满库）。

逻辑层 **`BuildingProduction` 已正确切换状态并广播 `OnStateChanged`**，但**玩法层几乎没有任何表现**——玩家「玩」的时候感受不到「为什么停」。

**代码证据 — 状态消费方仅 UI 文字**

全项目订阅 `OnStateChanged` 的仅有 **`BuildingStatusText`**：

| 层级 | 行为 |
|------|------|
| 逻辑 | `BuildingProduction.ChangeState` → `OnStateChanged.Invoke(state)` |
| UI | `BuildingStatusText` 改 TMP 为 `Need N1x2(has 0)` / `Output full (8/8)` |
| **玩法/视觉** | **无** — 无建筑材质/颜色、无动画停转、无音效、无仓库高亮、无屏幕提示 |

**具体缺失（Primitive 项目也做得到，无需美术资源）**

| 停产原因 | 逻辑有 | 玩法层应有（示例） | 当前有 |
|----------|--------|-------------------|--------|
| InputMissing | ✅ | 建筑/Input 区变色（红）、闪烁 Input 仓库、Idle 动画停 | ❌ 仅文字 |
| OutputFull | ✅ | Output 区变色（黄）、Output 仓库堆叠顶满提示 | ❌ 仅文字 |
| Producing 恢复 | ✅ | 恢复默认色、继续生产 tick 的可视节奏 | ❌ 文字切回建筑名 |

**为何评审认为「结果 questionable」**

- 题面要求「**游戏机制**能够正常运行」；机制不仅 = 数据正确，还包括 **玩家可感知、可操作的反馈闭环**。
- 停产时 `Update` 仍跑计时器，建筑外观与正常运行时一致；新玩家靠近建筑**看不出**是在等料还是已满。
- `BuildingStatusText` 为世界空间小字 + 英文调试风格文案，更像 **开发者 HUD**，不是 gameplay design。

**合理改法（→ IMP-14）**

新增 **`BuildingProductionView`**（或扩展现有 View），订阅 `OnStateChanged`：

- 状态 → 建筑根节点 **Renderer.material.color** 或 Emission（URP 可用 `_BaseColor`）
- 可选：Output 满时 **`WarehouseView` 最后一格脉冲**；缺料时 Input 区 **Collider 边框 Gizmo 式缩放**
- 文案保留 `BuildingStatusText`，但玩法反馈不依赖「玩家读字」

**与架构文档的关系**：`架构设计.md` 写了 UI 接入，但只覆盖 TMP；需在 v1.5 补充「状态 → 玩法反馈」数据流。

---

### 反馈总结：为何这三条直指「架构负责人」

| 反馈 | 暴露的能力缺口 |
|------|----------------|
| 库存重复 | 不会/未做 **领域建模与复用**（Domain layer） |
| Singleton 分散 | **依赖生命周期管理**意识不足；IMP-11 治标不治本 |
| 停产无玩法反馈 | **逻辑与体验闭环**断裂；有状态机但未完成「可玩性交付」 |

这三条与「画面精不精美」无关，恰恰是 **代码架构 + 机制体验** 的核心——与题面评分维度一致，且比「没勾验证清单」更能解释「不适合 solo 定核心」。
