# 多会话并行执行计划

> 本文档将架构设计拆分为多个独立的 AI 会话任务，支持并行执行。
> 每个会话对应一个 Claude Code 窗口，所有会话操作同一项目目录。

---

## 一、并行策略

```
阶段 0（串行，必须先完成）
├── 产出所有枚举 + SO 数据类 + 公共接口定义
│
阶段 1（3 会话并行，阶段 0 完成后同时启动）
├── 会话 A: 建筑与仓库
├── 会话 B: 玩家系统
├── 会话 C: 资源传输动画
│
阶段 2（串行，阶段 1 全部完成后）
└── 场景搭建 + 集成联调
```

---

## 二、文件冲突分析

### 无冲突区域（不同目录）

各会话写入的目录完全不重叠：

| 会话 | 写入目录 |
|------|----------|
| A | `Assets/Scripts/Building/` |
| B | `Assets/Scripts/Player/` |
| C | `Assets/Scripts/Transfer/` |

### 潜在冲突点与解决方案

| 冲突场景 | 风险 | 解决方式 |
|----------|------|----------|
| 多个会话修改 `.csproj` / `.sln` | **无冲突** — Unity 自动重新生成，无需手动编辑 | 不要手动编辑 `.csproj` 或 `.sln` |
| 多个会话创建 `.meta` 文件 | **无冲突** — Unity 导入时自动生成，文件名不重叠 | 不要在会话中手动创建 `.meta` |
| 场景文件 `SampleScene.unity` | **严重冲突** — 阶段 1 不要动场景 | 阶段 2 由主会话独自操作场景 |
| SO 资产文件（`.asset`） | **可能冲突** — 阶段 0 统一创建，阶段 1 只读 | 阶段 1 所有会话只能读取 SO，不得创建/修改 |
| 同一个 `.cs` 文件被两个会话编辑 | **严重冲突** — 绝对禁止 | 严格按文件分配表执行，不可越界 |
| Git 合并冲突 | **不会发生** — 所有会话在同一工作目录操作，无分支 | N/A |

## 三、阶段 0: 基础层（主会话，必须先完成）

**目标**：产出所有枚举、SO 数据类、SO 资产文件、以及跨会话依赖的公共接口定义。

> 阶段 0 未完成前，阶段 1 的所有会话无法启动（编译都过不了）。

### 步骤

1. 创建目录结构：
   - `Assets/Scripts/Core/`
   - `Assets/Scripts/Data/`
   - `Assets/Scripts/Building/`
   - `Assets/Scripts/Player/`
   - `Assets/Scripts/Transfer/`
   - `Assets/Scripts/Util/`



## 四、阶段 1: 并行实现

### 公共约束（所有会话必须遵守）

1. **只能创建/编辑分配给你的 `.cs` 文件**，不得修改其他会话的文件
2. **不得修改场景文件**（`.unity`）
3. **不得创建或修改 SO 资产**（`.asset`），只能读取
4. **不得修改 `Core/` 和 `Data/` 下的文件**（阶段 0 已产出）
5. 依赖其他会话的类时，假设其 **public API 已存在**（按下方接口约定编写），编译错误在阶段 2 解决
6. 代码注释使用中英双语
7. 所有 `MonoBehaviour` 的 `[SerializeField]` 引用留空，阶段 2 统一拖拽绑定

### 会话 A: 建筑与仓库

**产出文件**：
- `Assets/Scripts/Building/Warehouse.cs`
- `Assets/Scripts/Building/BuildingProduction.cs`
- `Assets/Scripts/Building/WarehouseView.cs`

**依赖**（只读）：
- `ResourceType.cs`、`WarehouseType.cs`
- `ResourceDatabase.cs`
- `BuildingProductionConfig.cs`

**Warehouse.cs 接口约定**（供会话 B/D 引用）：
```csharp
public class Warehouse : MonoBehaviour
{
    [SerializeField] private int capacity = 10;  // 仓库自身容量，Inspector 中配置
    public int Capacity { get; }                 // 对外只读
    public WarehouseType warehouseType;
    public Dictionary<ResourceType, int> StoredResources { get; }
    public Transform contentPoint;   // 资源 Lerp 目标点
    public Transform spawnPoint;     // 建筑产出点（仅 Output 使用）

    public bool CanAddResource(ResourceType type, int amount);
    public bool CanRemoveResource(ResourceType type, int amount);
    public void AddResource(ResourceType type, int amount);
    public void RemoveResource(ResourceType type, int amount);
    public int GetTotalStored();
    public bool IsFull();
    public bool HasResources(ResourceType type, int amount);

    public UnityEvent<ResourceType, int> OnResourceAdded;
    public UnityEvent<ResourceType, int> OnResourceRemoved;
}
```

**BuildingProduction.cs 接口约定**：
```csharp
public class BuildingProduction : MonoBehaviour
{
    public BuildingProductionConfig config;
    public BuildingState CurrentState { get; }
    public UnityEvent<BuildingState, string> OnStateChanged;
}
```

**实现要点**：
- `Warehouse.cs`：维护 `Dictionary<ResourceType, int>` 库存与容量检查；**不**处理 Trigger（由玩家侧 `PlayerWarehouseInteraction` 负责）
- `BuildingProduction.cs`：`Update()` 中计时器 + 状态机（Producing / InputMissing / OutputFull），通过 `GetComponentsInChildren<Warehouse>()` 查找子物体的 Input/Output 仓库；产出前按 `outputRecipe` 检查空位

### 会话 C: 资源传输动画

**产出文件**：
- `Assets/Scripts/Transfer/TransferManager.cs`
- `Assets/Scripts/Transfer/ResourceObjectPool.cs`

**依赖**（只读）：
- `ResourceType.cs`
- `ResourceDatabase.cs`
- `ResourceTransferConfig.cs`

**TransferManager.cs 接口约定**（单例，供会话 A/B 调用）：
```csharp
public class TransferManager : MonoBehaviour
{
    public static TransferManager Instance { get; }

    // 立即播放传输动画（供 BuildingProduction 调用）
    // Play transfer immediately (called by BuildingProduction)
    public void PlayTransfer(ResourceType type, Transform startPoint, Transform endPoint, System.Action onComplete);

    // 排队传输（供 PlayerWarehouseInteraction 调用）
    // Enqueue transfer request (called by PlayerWarehouseInteraction)
    public void EnqueueTransfer(ResourceType type, Transform startPoint, Transform endPoint, System.Action onComplete);
}
```

**ResourceObjectPool.cs 接口约定**（单例，全系统共用）：
```csharp
public class ResourceObjectPool : MonoBehaviour
{
    public static ResourceObjectPool Instance { get; }

    public void Prewarm(GameObject prefab, int count);
    public GameObject Get(GameObject prefab, Vector3 position, Quaternion rotation, Transform parent = null);
    public void Return(GameObject obj);
}
```

**实现要点**：
- `TransferManager.cs`：
  - 单例模式，组合动画播放与队列调度
  - `PlayTransfer()`：从 ResourceObjectPool 取实例，用 `StartCoroutine` 做抛物线和 Lerp 动画（弧高和曲线从 `ResourceTransferConfig` 读取），到达终点后归还到池
  - `EnqueueTransfer()`：追加到内部队列，`Update()` 中按 `transferInterval` 间隔逐个出队调用 `PlayTransfer()`
- `ResourceObjectPool.cs`：
  - 中心化对象池，维护 `Dictionary<GameObject, Queue<GameObject>>`（prefab → 闲置实例）
  - 按 prefab 自动分类管理，未知来源物体自动销毁

### ResourceTransferConfig 字段（实际实现）

| 字段 | 类型 | 说明 |
|------|------|------|
| moveSpeed | float | 移动速度 |
| moveCurve | AnimationCurve | 移动曲线（默认线性） |
| initialPoolSize | int | 对象池初始大小 |
| transferInterval | float | 队列中每个传输请求的间隔（秒） |
| arcHeight | float | 抛物线弧高（Y 轴偏移峰值） |

## 五、阶段 2: 集成联调（主会话）

**前置条件**：阶段 1 的全部会话完成，代码无编译错误。

### 步骤

1. **修复编译错误**：检查所有跨会话引用，确保命名和方法签名一致
2. **创建 Prefab**：
   - 建筑 Prefab（Cube + BuildingProduction + 两个子物体 Warehouse）
   - 建筑 1/2/3 Prefab Variant（挂不同 Config）
   - 仓库 Prefab（Cube + Warehouse + WarehouseView + Collider isTrigger）
   - 玩家 Prefab（Capsule + CharacterController + PlayerController + PlayerBackpack + BackpackView + PlayerWarehouseInteraction + Trigger Collider）
   - 场景根节点：TransferManager + ResourceObjectPool（单例服务）
   - 资源方块 Prefab（三种颜色的 Cube Prefab，视觉表现由 Prefab 自身决定）
3. **场景搭建**：
   - 摆放三座建筑
   - 放置玩家起始点 + 虚拟摇杆 UI Canvas
   - 创建 EventSystem
4. **绑定引用**：
   - 每个建筑的 BuildingProduction 拖入对应 Config
   - BuildingProduction 的 Input/Output Warehouse 拖入对应子物体
   - 所有 SerializeField 引用填充
5. **Layer 配置**：
   - 创建 `Player` Layer 和 `Warehouse` Layer
   - Physics 矩阵中确保两者可碰撞触发
6. **调参测试**：按验证清单逐项检查

## 七、风险与注意事项

| 风险 | 缓解措施 |
|------|----------|
| 会话间 API 不一致 | 严格按照本文档的接口约定实现，方法签名不得自行修改 |
| Unity 热重载导致锁定 | 关掉 Unity 编辑器再启动并行会话，或使用 `EnterWorktree` 隔离 |
| 阶段 0 遗漏依赖 | 阶段 0 完成后做一次编译检查，确保所有 SO 类和枚举通过编译 |
| Git 工作树方案 | 如果需要更彻底的隔离，每个会话可 `EnterWorktree` 到独立工作树，完成后合并 |
| 串行瓶颈 | 阶段 0 约 30 分钟，阶段 1 三个会话可完全并行，阶段 2 约 30-60 分钟 |
