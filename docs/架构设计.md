# 资源生产链游戏 — 架构设计

> 版本: v1.3 | 引擎: Unity 6000.3.13f1 (LTS)

---

## 一、任务概述

三座建筑串联生产三种资源（N1 → N2 → N3），玩家通过虚拟摇杆控制角色，靠近仓库自动拾取/存放资源。核心考核点在于模块解耦、可扩展的架构设计。

## 三、数据层设计

### 3.1 ResourceType

```csharp
public enum ResourceType { N1, N2, N3 }
```

### 3.2 WarehouseType

```csharp
public enum WarehouseType { Input, Output }
```

- Input 仓库：只进不出。玩家将资源放入，建筑生产时消耗。
- Output 仓库：只出不进。建筑产出后放入，玩家从中拾取。

### 3.3 ResourceDatabase (ScriptableObject)

一张总表，定义所有资源类型的元数据。视觉表现由 Prefab 自身携带（无需单独配置材质和颜色）：

| 字段 | 类型 | 说明 |
|------|------|------|
| type | ResourceType | 资源枚举值 |
| displayName | string | 显示名称（如"铁矿石"） |
| transferPrefab | GameObject | 飞行中显示的小方块 Prefab |
| backpackPrefab | GameObject | 静态堆叠显示块 Prefab（背包与仓库 View 共用） |

### 3.4 BuildingProductionConfig (ScriptableObject)

每种建筑一个资产，定义生产配方（仓库容量属于 Warehouse 组件，不在此配置）：

| 字段 | 类型 | 说明 |
|------|------|------|
| buildingName | string | 建筑名称 |
| productionInterval | float | 生产间隔（秒） |
| inputRecipe | RecipeEntry[] | 每轮消耗的资源列表 |
| outputRecipe | RecipeEntry[] | 每轮产出的资源列表 |

`RecipeEntry` 结构：`{ ResourceType type; int amount; }`

**三座建筑的配方数据：**

| 建筑 | 消耗 | 产出 | 间隔 |
|------|------|------|------|
| 建筑 1 | — | N1 × 1 | 3s |
| 建筑 2 | N1 × 2 | N2 × 1 | 5s |
| 建筑 3 | N1 × 1 + N2 × 1 | N3 × 1 | 8s |

> 仓库容量在 Warehouse 组件的 `capacity` 字段上独立配置，不在配方 SO 中。典型值：建筑1 输入/输出=0/10，建筑2=8/8，建筑3=6/6。

### 3.5 ResourceTransferConfig (ScriptableObject)

| 字段 | 类型 | 说明 |
|------|------|------|
| moveSpeed | float | 飞行移动速度 |
| moveCurve | AnimationCurve | 移动曲线（默认线性） |
| initialPoolSize | int | 对象池初始大小 |
| transferInterval | float | **队列**出队间隔（秒），控制 `EnqueueTransfer` 请求之间的播放间隔 |
| arcHeight | float | 抛物线弧高 |

> **传输间隔说明**：`ResourceTransferConfig.transferInterval` 控制 TransferManager 内部队列的出队节奏；`PlayerWarehouseInteraction.transferInterval` 控制玩家靠近仓库时**发起**传输请求的节奏。两者职责不同，分别调参。

## 四、组件关系与数据流

```
BuildingProduction (MonoBehaviour)
├── Config (BuildingProductionConfig)  ← 拖入 SO
├── inputWarehouse: Warehouse          ← Awake 时从子物体查找
├── outputWarehouse: Warehouse
├── transferManager: TransferManager   ← 可选，留空用单例
├── State: enum { Producing, InputMissing, OutputFull }
├── OnStateChanged: UnityEvent<BuildingState>
│   └── UI（BuildingStatusText）自行查仓库+配方格式化原因
│
└── 生产 Tick / 恢复:
    1. 检查 inputRecipe 每种资源在 inputWarehouse 是否足够
       不足 → State = InputMissing，跳过本轮
    2. 检查 outputWarehouse 是否能容纳 outputRecipe 全部产出
       不足 → State = OutputFull，跳过本轮
    3. 同步扣除 inputWarehouse 消耗资源（消耗动画纯视觉）
    4. 播放产出动画，到达后回调向 outputWarehouse 添加资源
    5. 通过 TransferManager.PlayTransfer() 播放动画
    6. TryResumeProduction()（Input 入库 / Output 取货后立即）→ 条件满足则立刻 TryProduce()
```

### Warehouse (MonoBehaviour)

```
├── warehouseType: WarehouseType
├── capacity: int                       ← [SerializeField] 仓库自身决定容量
├── storedResources: Dictionary<ResourceType, int>
├── acceptedTypes: List<ResourceType>   ← 空列表=接受全部类型
├── OnResourceAdded / OnResourceRemoved ← 供 WarehouseView 订阅
├── [SerializeField] Transform contentPoint   ← 资源 Lerp 起点/终点
└── [SerializeField] Transform spawnPoint     ← 建筑产出点（仅 Output 仓库使用）
```

> Trigger 检测由 **PlayerWarehouseInteraction**（玩家侧）负责，Warehouse 本身不处理碰撞。

### BuildingStatusText (MonoBehaviour)

```
├── building: BuildingProduction       ← 可选，留空则从父级查找
├── TMP_Text（RequireComponent）        ← 支持 TextMeshProUGUI；初始 text 作为建筑名
└── 行为:
    Producing     → 显示建筑名
    InputMissing  → 建筑名 + 缺料详情（查 inputRecipe + InputWarehouse）
    OutputFull    → 建筑名 + 满库计数（查 OutputWarehouse，监听取货刷新）
    库存已够但仍为 InputMissing → 第二行显示 "Ready"（过渡文案）
```

**与 BuildingProduction 的协作：**

| 事件来源 | BuildingProduction | BuildingStatusText |
|----------|-------------------|-------------------|
| 状态切换 | `OnStateChanged` 广播 | 订阅并刷新全文 |
| Input 入库 | `TryResumeProduction()` 可能切回 Producing | 订阅 `InputWarehouse.OnResourceAdded` 刷新缺料计数 |
| Output 取货 | `TryResumeProduction()` 可能恢复生产 | 订阅 `OutputWarehouse.OnResourceRemoved` 刷新满库计数 |

> `OnStateChanged` **仅传递** `BuildingState` 枚举，不传原因字符串；原因由 UI 查仓库+配方自行格式化。

### 交互流程

```
玩家靠近 Output 仓库 → PlayerWarehouseInteraction.OnTriggerEnter 检测
→ 按 transferInterval 从仓库取出资源（立即扣库存）
→ TransferManager.EnqueueTransfer(资源, 仓库contentPoint, 角色背上, 回调: 背包+1)
→ 回调失败时资源退回仓库

玩家靠近 Input 仓库 → PlayerWarehouseInteraction 检测
→ 按 transferInterval 从背包取出资源（立即扣背包）
→ TransferManager.EnqueueTransfer(资源, 角色背上, 仓库contentPoint, 回调: 仓库+1)
→ 回调失败时资源退回背包
```

### 四种传输动画

| 场景 | 起点 | 终点 | 触发时机 | API |
|------|------|------|----------|-----|
| 建筑 → 仓库 | spawnPoint | contentPoint | 生产完成时 | `PlayTransfer()` |
| 仓库 → 角色 | contentPoint | 角色背上 | 玩家靠近 Output 仓库 | `EnqueueTransfer()` |
| 角色 → 仓库 | 角色背上 | contentPoint | 玩家靠近 Input 仓库 | `EnqueueTransfer()` |
| 仓库 → 建筑 | contentPoint | 建筑消耗点 | 建筑消耗资源时 | `PlayTransfer()` |

四种场景统一由 `TransferManager` 处理，仅传入不同的起点/终点/回调参数。

### 库存与动画时序

| 操作 | 库存变更时机 | 动画角色 |
|------|-------------|----------|
| 建筑消耗输入 | 生产开始时**立即扣除** | 纯视觉 |
| 建筑产出 | 动画到达后**回调添加** | 数据与视觉同步 |
| 玩家拾取 | 发起时**立即从仓库扣除** | 到达后回调加入背包 |
| 玩家存放 | 发起时**立即从背包扣除** | 到达后回调加入仓库 |

回调中再次校验目标容器容量；失败时将资源退回来源容器并输出 Warning 日志。

## 五、关键设计决策

| 决策 | 理由 |
|------|------|
| WarehouseType 枚举收敛行为 | Output 只出不进，Input 只进不出，避免交互歧义 |
| 配方用列表 | 建筑 4 消耗 N2+N3 生产 N4 时无需改代码 |
| 资源视觉与配方分离 | 换皮肤只改 ResourceDatabase，不动配方数据 |
| 一建筑一 Config SO | 三人并行开发不冲突；Prefab Variant 只需换 SO |
| TransferManager 统一入口 | 四种传输场景本质相同，合并动画播放与队列调度 |
| 事件驱动 | 建筑模块不直接引用其他模块，状态变化通过 UnityEvent 抛出 |
| 对象池 | 频繁 Instantiate/Destroy 资源方块的开销可控，预留扩展点 |
| Model-View 分离 | Warehouse/PlayerBackpack 管数据，WarehouseView/BackpackView/BuildingStatusText 管显示 |
| BuildingProduction 不依赖 Animator/Rigidbody/Joint | 纯数据/逻辑驱动，便于测试和维护 |

## 六、实现顺序

1. **数据层** — 枚举、SO 类、生成配置资产
2. **仓库** — Warehouse.cs 实现库存管理
3. **建筑生产** — BuildingProduction.cs 状态机与生产循环
4. **资源传输** — TransferManager + ResourceObjectPool
5. **玩家系统** — 移动、背包、交互、View
6. **场景搭建** — 布置建筑、玩家、调参

## 八、建筑状态 UI 接入（编辑器）

> **需你在编辑器中操作** — 为每座建筑挂载世界空间状态文字。完整步骤亦见 `README.zh-CN.md` 的「建筑状态 UI 接入」章节。

### 推荐层级

```
BuildingRoot（BuildingProduction）
└── StatusCanvas（Canvas · World Space · Event Camera = Main Camera）
    └── StatusText（TextMeshProUGUI + BuildingStatusText）
```

### 步骤摘要

1. 建筑根下创建 **World Space Canvas**，Scale 约 `0.01`，置于建筑上方。
2. 子物体创建 **Text - TextMeshPro**，**Text** 填建筑名（正常生产时显示）。
3. 添加 **BuildingStatusText**；`Building` 拖入根节点 `BuildingProduction`（或留空且保持子层级关系）。
4. Apply Prefab，Play 验证：缺料 → `Need N1x2(has 0)`；满库 → `Output full (n/cap)`；恢复 → 仅建筑名。

### 自定义 UI

订阅 `BuildingProduction.OnStateChanged`，在回调中根据 `BuildingState` 更新自有 UI；缺料/满库详情需自行读取 `InputWarehouse` / `OutputWarehouse` 与 `Config.inputRecipe`。
